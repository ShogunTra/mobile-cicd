# Введение в CI/CD в мобильной разработке (Android / Java)

В современной разработке программного обеспечения важны не только качество кода, но и скорость его доставки пользователям. Для решения этой задачи применяется подход **CI/CD (Continuous Integration / Continuous Delivery)** — набор практик и инструментов, автоматизирующих проверку, сборку, тестирование и выпуск программного обеспечения.

Для мобильных приложений автоматизация особенно важна. Если веб-сервис можно обновить на сервере за несколько секунд, то мобильное приложение необходимо:

* скомпилировать;
* протестировать;
* подписать криптографическим ключом;
* подготовить релизный пакет;
* доставить пользователям через магазин приложений.

CI/CD позволяет выполнять все эти операции автоматически и воспроизводимо.

---

# 1. Что такое CI/CD

## Continuous Integration (CI) — Непрерывная интеграция

**Continuous Integration** — практика регулярного объединения изменений от разных разработчиков в общий репозиторий.

Каждый коммит или запрос на слияние (**Pull Request / Merge Request**) запускает автоматический процесс проверки проекта.

Типичный процесс CI включает:

1. Получение исходного кода из репозитория.
2. Установку зависимостей.
3. Статический анализ кода.
4. Компиляцию проекта.
5. Запуск автоматических тестов.
6. Формирование отчёта о результате.

Основная цель CI — обнаружить ошибки как можно раньше, пока их исправление обходится дешево.

> Принцип CI: код не считается готовым к интеграции, пока автоматические проверки не завершились успешно.

---

## Continuous Delivery (CD) — Непрерывная доставка

**Continuous Delivery** является продолжением CI.

После успешного прохождения всех проверок система автоматически подготавливает релиз:

* собирает приложение;
* подписывает его релизным сертификатом;
* создаёт готовый пакет распространения;
* публикует его во внутреннее хранилище или систему тестирования.

При этом окончательное решение о публикации в магазин приложений принимает человек.

### Пример

Разработчик создаёт новый релиз.

CI/CD автоматически:

* собирает Android App Bundle;
* подписывает его;
* загружает в тестовый канал Google Play.

После проверки QA-инженер нажимает кнопку публикации.

---

## Continuous Deployment — Непрерывное развертывание

**Continuous Deployment** идёт ещё дальше.

Каждое изменение, успешно прошедшее все проверки, автоматически публикуется без участия человека.

Схема выглядит так:

```text
Commit
   ↓
Tests
   ↓
Build
   ↓
Publish
   ↓
Users
```

Для мобильных приложений такой подход используется редко из-за:

* требований модерации магазинов приложений;
* рисков публикации критических ошибок;
* необходимости ручного контроля релизов.

Поэтому в мобильной разработке чаще применяется **Continuous Delivery**, а не **Continuous Deployment**.

---

# 2. Архитектура мобильного CI/CD-конвейера

Для Android-проектов на Java или Kotlin система сборки обычно основана на **Gradle**.

Локально разработчик запускает команды через Gradle Wrapper:

```bash
./gradlew assembleDebug
./gradlew test
./gradlew bundleRelease
```

CI-сервер выполняет те же действия в изолированной среде.

Типичный конвейер выглядит следующим образом:

```text
Git Push
    ↓
Code Quality
    ↓
Unit Tests
    ↓
UI / Integration Tests
    ↓
Build
    ↓
Signing
    ↓
Artifact Storage
    ↓
Distribution
```

# 3. Этапы мобильного CI/CD

## Этап 1. Проверка качества кода

Перед сборкой выполняется статический анализ исходного кода.

### Инструменты

* Android Lint
* Checkstyle
* PMD
* SpotBugs
* Detekt (для Kotlin)

### Что проверяется

* ошибки стиля;
* потенциальные утечки памяти;
* неиспользуемый код;
* нарушение архитектурных правил;
* устаревшие API;
* проблемы безопасности.

Преимущество статического анализа заключается в том, что многие ошибки обнаруживаются ещё до запуска приложения.

---

## Этап 2. Автоматическое тестирование

После успешного анализа запускаются тесты.

### Unit-тесты

Проверяют бизнес-логику приложения без запуска Android-системы.

Пример:

```java
assertEquals(5, calculator.divide(10, 2));
```

Unit-тесты выполняются быстро и обычно занимают несколько секунд.

---

### Instrumented-тесты

Запускаются внутри Android-среды.

Для этого CI-сервер:

1. создаёт Android Emulator;
2. устанавливает приложение;
3. выполняет тестовые сценарии;
4. проверяет результат.

Пример сценария:

```text
Нажать "5"
Нажать "+"
Нажать "5"
Нажать "="
Проверить отображение "10"
```

Для UI-тестирования часто используются:

* Espresso;
* UI Automator;
* Appium.

---

## Этап 3. Сборка приложения

После успешного прохождения тестов запускается сборка.

Для релизной версии обычно используется команда:

```bash
./gradlew bundleRelease
```

На этом этапе:

1. Java/Kotlin-код компилируется в `.class`.
2. Android Compiler (D8) преобразует его в `.dex`.
3. Оптимизатор R8 удаляет неиспользуемый код и уменьшает размер приложения.
4. Ресурсы объединяются в единый пакет.

Результатом является:

* **APK** — готовый установочный файл;
* **AAB (Android App Bundle)** — современный формат публикации в Google Play.

Сегодня для публикации в Google Play предпочтительным считается именно **AAB**.

---

## Этап 4. Подпись релиза

Android требует цифровой подписи каждого приложения.

Во время разработки используется временный отладочный ключ:

```text
debug.keystore
```

Для выпуска применяется отдельный релизный сертификат:

```text
release.keystore
```

Ключи не должны храниться в репозитории.

Обычно они размещаются в:

* GitHub Secrets;
* GitLab CI Variables;
* Jenkins Credentials;
* Vault-хранилищах.

Во время выполнения пайплайна секреты подставляются автоматически.

---

## Этап 5. Хранение артефактов

После сборки создаются артефакты.

Артефактами могут быть:

* APK;
* AAB;
* mapping-файлы ProGuard/R8;
* отчёты тестов;
* отчёты линтеров.

Они сохраняются в специальном хранилище CI-системы и могут быть скачаны в любой момент.

---

## Этап 6. Дистрибуция

Финальная стадия конвейера — распространение приложения.

### Для внутреннего тестирования

Наиболее популярные варианты:

* Firebase App Distribution;
* TestFlight (iOS);
* внутренние корпоративные порталы.

---

### Для пользователей

Релиз автоматически отправляется через API магазина приложений:

* Google Play
* RuStore
* Huawei AppGallery

После прохождения модерации обновление становится доступным пользователям.

# 5. Зачем мобильному разработчику CI/CD

| Без CI/CD                                 | С CI/CD                                        |
| ----------------------------------------- | ---------------------------------------------- |
| Сборка зависит от компьютера разработчика | Сборка выполняется в воспроизводимой среде     |
| Ошибки обнаруживаются поздно              | Ошибки выявляются сразу после коммита          |
| Можно забыть запустить тесты              | Тесты запускаются автоматически                |
| Возможны ошибки при подписи релиза        | Подпись выполняется по стандартному сценарию   |
| Релизы выпускаются вручную                | Выпуск частично или полностью автоматизирован  |
| Высокий риск человеческих ошибок          | Процесс становится предсказуемым и повторяемым |

---

# Заключение

CI/CD в мобильной разработке — это не просто автоматическая сборка приложения, а полноценный процесс контроля качества и доставки программного обеспечения.

Для Android-проектов типичный конвейер включает:

1. Проверку качества кода.
2. Автоматическое тестирование.
3. Сборку приложения.
4. Подпись релиза.
5. Хранение артефактов.
6. Дистрибуцию.

Использование CI/CD позволяет ускорить выпуск новых версий, уменьшить количество ошибок и сделать процесс разработки воспроизводимым независимо от конкретного компьютера разработчика.
